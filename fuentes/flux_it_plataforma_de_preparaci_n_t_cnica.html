Guía de Estudio Avanzada

Sintetizado directo de "Mi Granito de Java", principios de arquitectura distribuidora y bases teóricas de Flux IT.

1. Los Principios SOLID

Representan las mejores directrices para codificar de manera extensible, permitiendo que la arquitectura sea mantenible ante demandas imprevistas.

SRP (Single Responsibility) Una clase tiene solo una razón lógica para sufrir modificaciones. Evita controladores sobrecargados.
OCP (Open/Closed) Abierto a la extensión mediante polimorfismo, cerrado a la modificación de la clase base.
LSP (Liskov Substitution) Las subclases deben poder sustituir a su clase padre sin que falle el sistema. No alterar flujos esperados ni lanzar excepciones ajenas.
ISP (Interface Segregation) Muchos contratos específicos son preferibles a una sola interfaz monolítica gigante que arrastre métodos no utilizados.

2. Los Principios GRASP

A diferencia de SOLID, GRASP se enfoca en asignar responsabilidades y coordinar las interacciones de los objetos de dominio.

  • Experto en Información (Information Expert): Asigna la responsabilidad al objeto que tiene los datos requeridos para efectuar la acción (ej. el Carrito de compras calcula el total, no el PagoService).
  • Creador (Creator): Define qué clase es responsable de instanciar un objeto basándose en su composición de vida o uso estrecho.
  • Fabricación Pura (Pure Fabrication): Creación de clases artificiales cohesivas sin contraparte física en el dominio real, solo para salvar la modularidad del código (ej. DataExporter o Logger).
  • Controlador (Controller): Primer receptor no gráfico de llamadas externas que comanda el flujo de casos de uso del sistema.

3. Testing y Dobles de Prueba

El núcleo técnico de Flux IT está dominado por el TDD y la cobertura automatizada. Distinguir el rol de las dependencias duplicadas es fundamental:

Stub (Doble Pasivo): Provee datos estáticos controlados y preconfigurados para alimentar una consulta necesaria del test. No aserta llamadas ni flujos.

Mock (Doble Activo): Registra las interacciones de llamada internas. Valida aserciones sobre cuántas veces y con qué parámetros se invocó un servicio periférico.

4. Resiliencia, Event-Driven (EDA) y Cloud

Patrones determinantes para asegurar la escalabilidad horizontal infinita y tolerancia al colapso de infraestructuras complejas en nubes como AWS, Azure o GCP.

Saga Pattern Consistencia eventual a lo largo de transacciones distribuidas. Usa transacciones locales secuenciales y ejecuta acciones de reversa (Compensaciones) si ocurre algún error. Puede implementarse por Orquestación (con un bus controlador) o Coreografía (reactiva).
Circuit Breaker Protección de APIs que se abren (cortan solicitudes) al encadenar errores temporales en servicios terceros. Posee tres estados: Cerrado (pasa todo), Abierto (falla rápido localmente) y Semi-abierto (pruebas mínimas de salud).
Event-Driven Architecture (EDA) Sistemas que reaccionan de manera desacoplada a publicaciones de hechos inmutables (eventos) usando brokers como Apache Kafka o RabbitMQ. Aísla espacial y temporalmente al emisor del receptor.
Rate Limiting Regulación del tráfico entrante mediante algoritmos (Token Bucket, Leaky Bucket) para evitar caídas del core y garantizar el SLA ante ráfagas inesperadas de solicitudes.

5. Code Smells & Refactorización

"Mi Granito de Java" insiste mucho en la refactorización para sanar código podrido de forma estructurada. Aquí tienes los peores olores del software legacy:

Primitive Obsession Uso de tipos de datos primitivos nativos para encapsular reglas de negocio complejas (ej. guardar correos electrónicos como string puro sin validación). Solución: Reemplazar el valor con un Value Object dedicado.
Feature Envy (Envidia de Funciones) Un método que consulta intensamente los getters de una clase vecina para calcular su propia lógica. Solución: Aplicar "Move Method" para arrastrar la función a su dueño real.
