Guía de Estudio de Ingeniería de Software y Arquitectura de Sistemas

Subject: Preparación para Evaluación Técnica y Entrevistas (Flux IT) Topics:

  • Buenas Prácticas de Código y Principios de Diseño (SOLID, DRY, YAGNI, Cohesión/Acoplamiento)
  • Patrones de Diseño Clásicos (Creacionales, Estructurales y de Comportamiento) en Java, Node.js y Python
  • Olores de Código (Code Smells) y Técnicas de Refactorización Críticas (Reemplazar condicional con polimorfismo)
  • Los Principios GRASP y Patrones Corporativos JEE (DAO, DTO, Facade)
  • Estrategias de Testing Automatizado y Mocking (Unit Testing, Mocks vs Stubs, TDD)
  • Patrones de Resiliencia y Arquitectura Dirigida por Eventos (Circuit Breaker, Saga, Rate Limiting, Event-Driven/EDA)

Summary: Esta guía de estudio sintetiza los pilares fundamentales para el diseño, desarrollo y mantenimiento de software escalable, modular y tolerante a fallos, asimilando conceptos clave de la popular academia de diseño y del blog "Mi Granito de Java". Está estructurada para resolver de manera ágil exámenes de screening técnico (como TestGorilla) y proveer un marco conceptual impecable para discutir decisiones de arquitectura en sistemas de alto impacto (típicos de Flux IT en industrias como finanzas y seguros).

1. Pilares de Diseño y Código Limpio

  • Cohesión y Acoplamiento: La arquitectura ideal busca alta cohesión (que cada módulo tenga una responsabilidad única y muy enfocada) y bajo acoplamiento (independencia entre módulos para que los cambios en uno no impacten en otros).
  • Inyección de Dependencias (DI) vs Inversión de Dependencias (DIP): DIP es el principio de diseño abstracto de SOLID (los módulos de alto nivel no deben depender de los de bajo nivel, ambos deben depender de abstracciones). La Inyección de Dependencias (DI) es un patrón de diseño técnico que implementa DIP pasando las dependencias desde afuera (usualmente por constructor).
  • Polimorfismo: Pilar de la programación orientada a objetos (POO) que permite a distintos objetos responder al mismo mensaje o firma de método de maneras completamente diferentes de acuerdo a su implementación concreta.
  • SOLID (Conceptos Rápidos):
    • SRP (Responsabilidad Única): Una clase debe hacer solo una cosa (un solo actor/razón para cambiar).
    • OCP (Abierto/Cerrado): Extensible sin modificar el archivo original. Se logra programando interfaces o clases abstractas.
    • LSP (Sustitución de Liskov): Las subclases deben poder sustituir a sus clases base sin romper el programa. No lanzar excepciones imprevistas o restringir comportamientos.
    • ISP (Segregación de Interfaces): Es mejor tener muchas interfaces delgadas y específicas que una sola interfaz gigantesca que obligue a implementar métodos vacíos.
    • DIP (Inversión de Dependencias): Depender de abstracciones (interfaces), nunca de implementaciones concretas.

2. Los Principios GRASP (General Responsibility Assignment Software Patterns)

A diferencia de SOLID, GRASP se centra en la asignación básica de responsabilidades de las clases en el modelado de dominio. Es sumamente evaluado en entrevistas presenciales de Flux IT:

  • Experto en Información (Information Expert): ¿Quién debe tener la responsabilidad de calcular o hacer una acción? Aquella clase que tenga la información necesaria para hacerlo (ej. el Carrito debe calcular el total, no el ServicioDePago).
  • Creador (Creator): ¿Quién debe encargarse de instanciar la clase B? La clase A si: A contiene o agrega a B, usa estrechamente a B o tiene los datos para inicializar a B.
  • Controlador (Controller): El primer objeto más allá de la capa de interfaz gráfica que recibe y coordina un mensaje del sistema (ej. controla el flujo del caso de uso).
  • Bajo Acoplamiento y Alta Cohesión: Principios evaluadores permanentes para mantener la modularidad y facilidad de cambio del sistema.
  • Polimorfismo: Asignar responsabilidades de comportamiento variable a tipos polimórficos en lugar de usar condicionales.
  • Fabricación Pura (Pure Fabrication): Crear una clase altamente cohesiva que no represente un concepto real delheader del dominio, sino que se cree únicamente para mantener el acoplamiento bajo y la cohesión alta (ej. un DataExporter o Logger).
  • Indirección (Indirection): Asignar la responsabilidad a un objeto intermediario para evitar el acoplamiento directo entre otros dos componentes (ej. un Adapter o un Broker).
  • Variaciones Protegidas (Protected Variations): Proteger el sistema de cambios externos envolviendo los puntos de inestabilidad con una interfaz estable.

3. Code Smells (Olores de Código) y Refactorización

Un fuerte de la escuela de "Mi Granito de Java". Detectar problemas en el código legacy te da un perfil técnico extremadamente pulido:

  • Switch Statements (Condicionales Múltiples): Estructuras condicionales enormes que toman decisiones según el tipo de objeto.
    • Refactorización: Replace Conditional with Polymorphism. Crear subclases polimórficas o aplicar los patrones Strategy o State.
  • Feature Envy (Envidia de Funciones): Un método que accede más a los getters de otro objeto que a los métodos de su propia clase.
    • Refactorización: Move Method (Mover el método) a la clase donde están los datos que se codician.
  • Primitive Obsession (Obsesión Primitiva): Usar tipos de datos primitivos (String, int) para modelar conceptos del negocio (ej. guardar un Email en un String sin validar).
    • Refactorización: Replace Data Value with Object creando Value Objects dedicados con comportamiento y validación interna.
  • Long Method (Método Largo): Un método que supera las 15-20 líneas, haciendo sumamente difícil su testing y lectura.
    • Refactorización: Extract Method (Extraer método) delegando el flujo lógico a métodos privados cohesivos con nombres expresivos.

4. Patrones de Diseño Clásicos y JEE Corporativos

Facade / Fachada (Estructural)

Proporciona una interfaz unificada y simplificada sobre un conjunto complejo de subsistemas para ocultarle los detalles técnicos y flujos internos al cliente. Muy usado para simplificar servicios core heredados.

Proxy (Estructural)

Actúa como un intermediario o sustituto para controlar el acceso a otro objeto. Permite inyectar lógica de auditoría, control de accesos, seguridad de roles, o implementar lazy loading (carga perezosa) y caché.

Command / Comando (Comportamiento)

Encapsula una solicitud de acción dentro de un objeto con un único método (execute()). Facilita el encolamiento de tareas, la parametrización de clientes de forma dinámica y la implementación de operaciones reversibles (Undo/Redo).

Patrones JEE Clásicos:

  • DAO (Data Access Object): Separa la lógica de acceso a la base de datos de la lógica de negocio, encapsulando las consultas SQL o JPA.
  • DTO (Data Transfer Object): Objeto plano inmutable diseñado exclusivamente para transportar datos entre capas (o servicios mediante red), evitando exponer la entidad de persistencia real de la base de datos.
  • Business Delegate: Actúa como un intermediario para desacoplar la capa de presentación de los servicios de negocio distribuidos.

5. Estrategias de Testing y Mocking

  • El Ciclo TDD (Test-Driven Development):
    1. Red (Rojo): Escribir una prueba unitaria que falle para una funcionalidad aún no implementada.
    2. Green (Verde): Escribir el código mínimo indispensable para que la prueba pase exitosamente.
    3. Refactor (Refactorizar): Limpiar, optimizar y estructurar el código sin romper el comportamiento ya validado.
  • Diferencia Fundamental entre Mock y Stub:
    • Stub (Doble Pasivo): Sirve para alimentar al código bajo prueba con respuestas preconfiguradas fijas (ej. forzar que una llamada a base de datos devuelva un usuario falso). No hace aserciones sobre llamadas.
    • Mock (Doble Activo): Simula el comportamiento pero se enfoca en verificar cómo se usó la dependencia (ej. asertar que el método enviarEmail() fue invocado exactamente una vez con los parámetros correctos).

6. Arquitectura de Resiliencia y Event-Driven (EDA)

  • Circuit Breaker (Cortocircuito): Patrón que protege llamadas a servicios externos inestables. Posees tres estados: Cerrado (peticiones fluyen normalmente), Abierto (intercepta las llamadas e inicia un retorno rápido fail-fast local), y Semi-abierto (peticiones mínimas de prueba).
  • Saga Pattern: Gestiona transacciones de negocio distribuidas mediante secuencias locales y transacciones compensatorias. Puede ser mediante Coreografía (coordinación asincrónica reactiva) o Orquestación (un orquestador central comanda).
  • Event-Driven Architecture (EDA): Comunicación asincrónica desacoplada impulsada por la publicación de hechos históricos inmutables (eventos).
    • Diferencia en DevOps: RabbitMQ maneja colas efímeras inteligentes (mensajes eliminados tras el ack), mientras que Apache Kafka funciona como un log persistente de eventos inmutables donde los consumidores gestionan de manera independiente su puntero de lectura (offset).


DevOps Reality Check: TestGorilla vs. La Entrevista en Flux IT

Es muy común (y de hecho es una excelente señal de tu seniority) que la sección de DevOps de TestGorilla te haya parecido "bastante obvia". Sin embargo, hay un doble filo en esto que tenés que conocer antes de rendir el examen oficial.

1. El "porqué" de la obviedad en TestGorilla

TestGorilla es una herramienta de screening masivo. No está diseñada para evaluar si sos un SRE especialista que puede configurar un clúster de Kubernetes de cero con los ojos cerrados, sino para filtrar candidatos que no tienen las bases mínimas.

Para un desarrollador con perfil de arquitectura o sólida experiencia en backend, preguntas sobre:

  • La diferencia entre COPY y ADD en Docker.
  • Qué es un Pod o una Liveness Probe en Kubernetes.
  • Un git stash o un git rebase.

Son conceptos del día a día. Si te parecieron obvios, significa que tu base conceptual está perfecta y que vas a pasar este filtro sin transpirar.

2. Las verdaderas "trampas" del test de DevOps

Aunque las preguntas sean conceptualmente sencillas, el examen real tiene dos factores de riesgo que no debes subestimar:

  • El tiempo (30 segundos por pregunta): Es el módulo más rápido de todos. No hay tiempo para interpretar enunciados ambiguos. Un error de lectura por apurado te puede costar un punto.
  • La semántica de los comandos: A veces meten preguntas con opciones muy parecidas sobre flags específicos de Git o Docker (ej. la diferencia exacta entre docker image prune y docker system prune -a, o cómo resolver un conflicto de rebase paso a paso).
Consejo para el test: Mantené la concentración al 100%. Que te parezca fácil no te haga bajar la guardia con el reloj.

3. La realidad en Flux IT (La entrevista técnica en vivo)

Acá es donde el panorama cambia drásticamente. Flux IT trabaja con clientes pesados del sector bancario y de seguros (como Banco Galicia o San Cristóbal Seguros), donde las migraciones a la nube y el uptime son críticos.

En la entrevista técnica con los arquitectos, esa "obviedad" de TestGorilla se transforma en problemas reales de diseño de infraestructura:

  1. Zero-Downtime Database Migrations:
    • La pregunta de Flux: "Si hacemos un despliegue Blue-Green, ¿cómo migramos el esquema de la base de datos SQL (por ejemplo, renombrar una columna) sin romper la versión vieja (Azul) que sigue activa recibiendo tráfico?" (Patrón Expand/Contract).
  2. Troubleshooting de K8s bajo presión:
    • La pregunta de Flux: "Tenés un pod en CrashLoopBackOff en el entorno de producción de un cliente de alta demanda y no tenés permisos para acceder a la consola de AWS. ¿Cómo investigás el problema usando únicamente kubectl?"
  3. Monitoreo y Alertas reales:
    • La pregunta de Flux: "Configuramos Prometheus y Grafana. ¿Qué métricas específicas (Golden Signals de SRE) elegirías para disparar una alerta al equipo de guardia ante una degradación del servicio de pagos?"

Resumen Estratégico

  • Para el TestGorilla: Sigue confiando en tu instinto. Tu nivel técnico es más que suficiente para pasar este bloque caminando. Solo cuida el reloj.
  • Para la entrevista con los líderes: Prepárate para justificar cómo tus decisiones de diseño de software (Fase 1 y 2) interactúan directamente con la infraestructura en la nube (Fase 3).

Candidato: Ezequiel

Foco: Sistemas de alta disponibilidad, entornos bancarios/seguros y resiliencia.

Escenario 1: Zero-Downtime Database Migrations (Expand/Contract)

El Problema en Producción

En arquitecturas monolíticas o de microservicios que comparten bases de datos relacionales, actualizar el esquema (ej. renombrar una columna, cambiar un tipo de datos de int a UUID, o dividir una tabla) típicamente requiere una ventana de mantenimiento (sistema caído) porque la versión activa del código fallaría al no encontrar la estructura vieja. Para clientes de Flux IT (como Banco Galicia), la ventana de mantenimiento de 00:00 a 04:00 hs ya no es viable para actualizaciones continuas.

La Solución: El Patrón Expand & Contract (Expandir y Contraer)

Es un patrón de migración de base de datos en fases que permite desacoplar los despliegues de base de datos de los despliegues de código de aplicación.

Fase 1 (Expand): [DB con Columna_Vieja] <--- Escribe/Lee ---> [App Versión 1]

                 [DB agrega Columna_Nueva] (Se replica la escritura en ambas)


Fase 2 (Transition): [DB] <--- Lee de Vieja / Escribe en ambas ---> [App Versión 2 (Blue-Green)]

                     * Se ejecuta un script en background para migrar datos históricos.


Fase 3 (Contract): [DB elimina Columna_Vieja] <--- Lee/Escribe en Nueva ---> [App Versión 3 (Verde)]


Paso a Paso de Ejecución Técnica:
  1. Fase de Expansión (Expand):
    • Se altera la base de datos para agregar la nueva columna o tabla (ej. email_nuevo), pero se conserva la vieja (email_viejo). El esquema se expande.
    • Se despliega la nueva versión de la aplicación. Esta versión debe ser tolerante al esquema dual: escribe en ambas columnas (email_viejo y email_nuevo) para asegurar la retrocompatibilidad, pero lee de la vieja.
  2. Fase de Sincronización:
    • Se ejecuta un job asincrónico (o un proceso batch liviano) que migra los datos históricos del campo viejo al nuevo únicamente para los registros antiguos que no fueron actualizados por la app en tiempo real.
  3. Fase de Transición:
    • Se despliega una actualización que cambia el puntero de lectura de la aplicación al nuevo campo (email_nuevo). Las escrituras continúan en ambos campos por precaución de rollback.
  4. Fase de Contracción (Contract):
    • Una vez verificado que el sistema está 100% estable y leyendo del nuevo campo, se elimina la columna vieja (email_viejo) del motor SQL. El esquema se contrae.

Escenario 2: Diagnóstico de K8s bajo restricción de privilegios

El Problema en Producción

En entornos bancarios hiper-regulados, los desarrolladores (e incluso los arquitectos) tienen accesos IAM sumamente restringidos debido al principio de mínimo privilegio. No tenés acceso a la consola de AWS (EKS), ni a logs centralizados (Splunk/Datadog) si el agente de logs está caído, ni permisos para entrar al nodo por SSH. Tu Pod está en CrashLoopBackOff.

Flujo de Triage Técnico usando únicamente kubectl:

  1. Paso 1: Inspección de Metadatos del Pod
    kubectl describe pod <nombre-del-pod> -n <namespace>
    • Qué buscar: La sección Events al final del output. Si el pod fue matado por falta de memoria RAM por el kernel de Linux (OOMKilled), verás el evento OOMKilled: true o un código de salida Exit Code: 137 (la CPU le mandó un SIGKILL al proceso). Si falló la sonda de supervivencia, verás Liveness probe failed.
  4. Paso 2: Inspección de Logs Recientes
    kubectl logs <nombre-del-pod> -n <namespace>
    • Qué buscar: El stacktrace de inicialización de la app (ej. error de conexión de base de datos, credenciales de Spring Boot corruptas).
  7. Paso 3: El truco para pods caídos de forma cíclica (--previous) Cuando un pod entra en CrashLoopBackOff, Kubernetes lo reinicia constantemente. Al reiniciarse, el comando kubectl logs por defecto te muestra los logs del contenedor activo (que acaba de levantar y quizás aún no falló).
    • El comando de oro:
      kubectl logs <nombre-del-pod> -n <namespace> --previous

    • Este flag le indica al plano de control de Kubernetes que te traiga los logs del contenedor justo antes de que colapsara en el ciclo anterior. Es el 90% del diagnóstico de inicialización fallida.
  8. Paso 4: Auditoría de Eventos del Namespace Si el pod ni siquiera llega a escribir logs (se queda en Pending o ContainerCreating), el problema no es de tu app, es de la infraestructura del clúster:
    kubectl get events -n <namespace> --sort-by='.metadata.creationTimestamp'
    • Qué buscar: Errores de aprovisionamiento de volúmenes EBS (EKS StorageClass), o falta de recursos en los nodos (FailedScheduling: 0/5 nodes are available: 5 Insufficient cpu).

Escenario 3: SRE Golden Signals (Métricas de Alerta Críticas)

El Concepto Teórico

Basado en el libro de SRE de Google, para servicios altamente transaccionales (como una pasarela de pagos) no sirve alertar al equipo de guardia simplemente con "Consumo de CPU > 80%". La CPU puede estar alta porque el garbage collector de Java está optimizando memoria de forma normal. Debes monitorear y alertar sobre las 4 Señales de Oro (Golden Signals):

                  ┌─────────────────────────────────────────┐

                  │          FOUR SRE GOLDEN SIGNALS        │

                  └────┬───────────┬───────────┬────────┬───┘

                       │           │           │        │

                    Latency     Traffic     Errors  Saturation


  1. Latencia (Latency): El tiempo que tarda en resolverse una solicitud.
    • Métrica de alerta: El Percentil 95 o 99 (P95/P99) de los tiempos de respuesta de la API de pagos (ej. http_request_duration_seconds_bucket). No uses el promedio, ya que oculta "outliers" horribles (usuarios que tardan 30 segundos en pagar).
  2. Tráfico (Traffic): La demanda del sistema medida en peticiones por segundo (RPS).
    • Métrica de alerta: Caídas abruptas de tráfico en horarios pico (ej. caída de peticiones de pago a 0 RPS), lo que indica un bloqueo de red o caída del DNS general.
  3. Errores (Errors): La tasa de peticiones que fallan.
    • Métrica de alerta: Tasa de respuestas HTTP 5xx sobre el total de llamadas superiores al $1\%$ en una ventana de 5 minutos.
  4. Saturación (Saturation): Qué tan "lleno" está el servicio (recursos de hardware o software limitados).
    • Métrica de alerta: Exhaustión del pool de conexiones de base de datos (ej. HikariCP en Java) o hilos del servidor web (Tomcat Thread Pool Saturation). Al llegar al $100\%$, el sistema colapsa aunque la CPU esté al $10\%$.

Bonus: El Patrón Outbox (Muy consultado por Arquitectos de Flux IT)

El Problema en Microservicios

Supongamos que un usuario realiza una compra. Tu microservicio de orden debe:

  1. Guardar la orden en su base de datos PostgreSQL local.
  2. Publicar un evento OrdenCreada en Kafka para que el microservicio de envíos empiece a trabajar.

El riesgo de inconsistencia de datos: Si guardas en la DB pero Kafka está caído, el usuario pagó pero el envío jamás se procesará. Si publicas en Kafka primero pero la escritura en la DB falla por un rollback de transacción, procesas un envío de una orden que no existe en el sistema.

La Solución: Transactional Outbox Pattern

Desacopla la escritura en la base de datos de la publicación del mensaje usando una única transacción ACID local.

[App] ──(Transacción ACID)──> [DB Transaccional]

                               ├──> Tabla: ORDEN

                               └──> Tabla: OUTBOX (Inserta Evento)

                                       │

                                (CDC / Debezium)

                                       │

                                       ▼

                             [Apache Kafka Broker]


  1. Paso 1: En lugar de enviar el mensaje directamente a Kafka, la aplicación inserta el registro de la orden en la tabla ORDEN y al mismo tiempo inserta el evento del mensaje en una tabla auxiliar llamada OUTBOX dentro de la misma transacción de base de datos.
  2. Paso 2: Al estar en la misma transacción ACID local, se garantiza que o se guardan ambos registros, o no se guarda ninguno.
  3. Paso 3 (Message Relay): Un componente independiente (ej. un conector de CDC - Change Data Capture como Debezium, o un polling worker ligero) lee la tabla OUTBOX de forma secuencial, publica los eventos en Kafka, y una vez confirmado el envío (ACK), elimina o marca como procesados los registros de la tabla OUTBOX.

Glosario Técnico Avanzado: Preparación Mesa Técnica

Este glosario define y desglosa los conceptos teóricos y de infraestructura que suelen aparecer de forma recurrente tanto en el screening de TestGorilla como en la entrevista técnica en vivo con los arquitectos de Flux IT.

1. ¿Qué es el Teorema de CAP? (Teorema de Brewer)

El Teorema CAP establece que en cualquier sistema de almacenamiento de datos distribuido, es físicamente imposible garantizar simultáneamente más de dos de las siguientes tres propiedades:

  • Consistencia ($C$ - Consistency): Todos los nodos del sistema ven los mismos datos al mismo tiempo. Si escribo un dato en el Nodo A, una lectura inmediata en el Nodo B me debe devolver exactamente ese dato nuevo.
  • Disponibilidad ($A$ - Availability): Cada petición no fallida recibe una respuesta (que no sea un error), sin la garantía de que contenga la versión más reciente del dato.
  • Tolerancia a Particiones ($P$ - Partition Tolerance): El sistema continúa funcionando a pesar de que la red se rompa o pierda mensajes entre los nodos.

La regla de oro en redes distribuidas:

Dado que en el mundo real las redes fallan de forma inevitable (es decir, la partición $P$ va a ocurrir tarde o temprano), un arquitecto siempre debe elegir entre Consistencia ($C$) o Disponibilidad ($A$) ante una falla:

  • Si elegís Consistencia (Sistemas CP): Bloqueás o rechazás las peticiones de los usuarios en los nodos que quedaron aislados para garantizar que nadie lea un saldo viejo. (Ejemplo: Un banco tradicional).
  • Si elegís Disponibilidad (Sistemas AP): Permitís que los usuarios sigan operando, aceptando que temporalmente van a leer datos desactualizados hasta que la red se arregle y se propague el cambio (consistencia eventual). (Ejemplo: El feed de Twitter/X o Instagram).

2. ¿Qué es ACID?

Es el conjunto de características que garantizan que una transacción en una base de datos relacional (como PostgreSQL, Oracle o MySQL) sea completamente segura y confiable, incluso ante caídas de tensión o fallas del servidor:

  • Atomicidad (Atomicity): La transacción es una unidad indivisible ("todo o nada"). O se ejecutan con éxito todas las operaciones de la transacción, o no se aplica ninguna (ej. si falla la acreditación en la cuenta destino, se cancela el débito en la cuenta origen).
  • Consistencia (Consistency): Una transacción solo puede llevar a la base de datos de un estado válido a otro estado válido, respetando todas las reglas y restricciones de integridad.
  • Aislamiento (Isolation): Las transacciones que se ejecutan simultáneamente no pueden interferir entre sí. El resultado de una transacción es invisible para otras hasta que se confirma.
  • Durabilidad (Durability): Una vez que la transacción se confirmó (commit), sus efectos persisten en disco de forma permanente y no se perderán aunque el servidor se apague inmediatamente después.

3. ¿Qué es SRE? (Site Reliability Engineering)

SRE (Ingeniería de Confiabilidad del Sitio) es una disciplina y un rol que nació en Google. Consiste en aplicar mentalidad y prácticas de ingeniería de software a problemas de infraestructura y operaciones.

A diferencia del sysadmin tradicional que resolvía todo de forma manual, un SRE automatiza las tareas repetitivas (toil), diseña la resiliencia de la arquitectura, y define métricas para medir la confiabilidad del servicio en producción:

  • SLI (Service Level Indicator): La métrica real (ej. el $99\%$ de las peticiones tardan menos de $200\text{ ms}$).
  • SLO (Service Level Objective): El objetivo interno (ej. queremos mantener un SLI del $99.9\%$ este mes).
  • SLA (Service Level Agreement): El acuerdo legal con el cliente (ej. si el sistema se cae más de un $0.1\%$ del tiempo al año, el banco nos cobra una multa).

4. ¿Qué es CrashLoopBackOff en Kubernetes?

No es una falla en sí misma, sino un estado del Pod de Kubernetes que le indica al clúster lo siguiente: "Este pod intentó arrancar, colapsó inmediatamente por un error interno, lo volví a intentar, volvió a colapsar, y ahora estoy esperando un tiempo prudencial (Backoff) antes de volver a intentar levantarlo para no saturar los nodos".

Causas comunes en producción:

  • Falta una variable de entorno requerida por la aplicación (ej. la IP de la base de datos).
  • Fallo en la conexión inicial a la base de datos (tiempo de espera agotado / credenciales incorrectas).
  • Un error de sintaxis al inicializar el contenedor Docker.
  • El Pod se quedó sin memoria RAM física y el Kernel de Linux lo mató (OOMKilled: Exit Code 137).

5. ¿Qué significa Liveness Probe Failed?

Es una alerta interna de Kubernetes que indica que la Prueba de Supervivencia del contenedor falló.

Una Liveness Probe es un chequeo periódico (ej. un ping HTTP a /healthz cada 10 segundos) que Kubernetes le hace a tu contenedor para saber si el proceso sigue vivo. Si la prueba falla consecutivamente (porque el servidor web se congeló por un deadlock o hilos bloqueados), Kubernetes asume que el proceso está "zombi", destruye el contenedor de forma automática y levanta uno limpio en su lugar.

El Misterio de la Calculadora: ¿Por qué te dejarían usarla?

En la pantalla de instrucciones que viste en la imagen anterior, el texto dice:

"Puedes utilizar papel, bolígrafo y calculadora durante la evaluación..."

La realidad es que esta es una advertencia genérica por defecto de TestGorilla. La plataforma le muestra exactamente ese mismo mensaje a todos los candidatos del mundo, sin importar si rinden un examen de Finanzas, de Física o de Arquitectura de Software.

En tus tests técnicos de Flux IT no vas a tener que usar una calculadora. Al ser una prueba teórica y de ritmo ágil (30-50 segundos por pregunta), como mucho te podrían pedir calcular una disponibilidad simple relacionada con los "nueves" del SLA:

$$\text{Downtime Anual Permitido para } 99.9\% \approx 8.76 \text{ horas}$$$$\text{Downtime Anual Permitido para } 99.99\% \approx 52.56 \text{ minutos}$$

Pero esas cuentas, si aparecen, son de opción múltiple y se resuelven de un vistazo por sentido común técnico. No te preocupes por tener que resolver integrales o logaritmos en el test; no viene por ahí la mano.
